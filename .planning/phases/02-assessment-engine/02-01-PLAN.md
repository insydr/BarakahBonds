## Objective

Create the database schema for the Question Bank, Scholarly Citations, Assessments, and Responses. This establishes the data foundation for the entire Assessment Engine, supporting versioned questions, scholarly citations with Islamic references, and encrypted response storage.

## Tasks

### Task 1: Create Assessment Tables Migration

<read_first>
- supabase/migrations/001_initial_schema.sql
- .planning/phases/02-assessment-engine/02-CONTEXT.md (D-01, D-03, D-06, D-19, D-22)
</read_first>

<action>
Create `supabase/migrations/002_assessment_schema.sql` with the following tables:

1. **questions** table:
   - `id` UUID PRIMARY KEY
   - `section` TEXT NOT NULL CHECK (section IN ('deen', 'dunya', 'aila', 'nafs'))
   - `order_index` INTEGER NOT NULL
   - `question_text` TEXT NOT NULL
   - `question_type` TEXT NOT NULL CHECK (question_type IN ('likert', 'multiple_choice', 'open_text'))
   - `options` JSONB (for multiple_choice: array of options)
   - `citation_id` UUID REFERENCES citations(id)
   - `version` INTEGER DEFAULT 1
   - `active` BOOLEAN DEFAULT true
   - `scholar_verified` BOOLEAN DEFAULT false
   - `created_at`, `updated_at` timestamps

2. **citations** table:
   - `id` UUID PRIMARY KEY
   - `type` TEXT NOT NULL CHECK (type IN ('quran', 'hadith'))
   - `source` TEXT NOT NULL (e.g., "Sahih Bukhari", "Surah Al-Baqarah")
   - `reference` TEXT NOT NULL (e.g., "2:177", "Book 1, Hadith 1")
   - `arabic_text` TEXT
   - `translation` TEXT NOT NULL
   - `scholar_verified` BOOLEAN DEFAULT false
   - `created_at` timestamp

3. **assessments** table:
   - `id` UUID PRIMARY KEY
   - `user_id` UUID NOT NULL REFERENCES auth.users(id)
   - `status` TEXT NOT NULL DEFAULT 'not_started' CHECK (status IN ('not_started', 'in_progress', 'completed'))
   - `current_section` TEXT CHECK (current_section IN ('deen', 'dunya', 'aila', 'nafs'))
   - `started_at` TIMESTAMPTZ
   - `completed_at` TIMESTAMPTZ
   - `scores` JSONB (category scores after completion)
   - `flags` JSONB (red flags detected)
   - `created_at`, `updated_at` timestamps

4. **assessment_responses** table:
   - `id` UUID PRIMARY KEY
   - `assessment_id` UUID NOT NULL REFERENCES assessments(id) ON DELETE CASCADE
   - `question_id` UUID NOT NULL REFERENCES questions(id)
   - `response_value` JSONB NOT NULL (encrypted for sensitive data)
   - `responded_at` TIMESTAMPTZ DEFAULT NOW()
   - UNIQUE constraint on (assessment_id, question_id)

Include all indexes for performance, RLS policies for data isolation, and helper functions.
</action>

### Task 2: Add RLS Policies for Assessment Tables

<read_first>
- supabase/migrations/001_initial_schema.sql (for RLS pattern reference)
- .planning/phases/02-assessment-engine/02-CONTEXT.md (D-22)
</read_first>

<action>
Add RLS policies to `002_assessment_schema.sql`:

1. **questions** table:
   - All authenticated users can view active, scholar_verified questions
   - No user INSERT/UPDATE/DELETE (admin only via service role)

2. **citations** table:
   - All authenticated users can view scholar_verified citations
   - No user INSERT/UPDATE/DELETE (admin only)

3. **assessments** table:
   - Users can view only their own assessments
   - Users can insert their own assessment (one per user for MVP)
   - Users can update their own assessment
   - Partners CANNOT view each other's assessments (privacy)

4. **assessment_responses** table:
   - Users can view only their own responses (via assessment ownership)
   - Users can insert responses for their own assessment
   - Partners CANNOT view individual responses

Add helper functions: `get_user_assessment_id()`, `has_completed_assessment()`.
</action>

<acceptance_criteria>

- File contains policies restricting assessment access to owner only
